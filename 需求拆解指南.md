# 产品经理需求拆解工作流程指南
## 基于原型图的康养钻石湾项目需求提取方法

## 一、需求拆解总体流程

### 1. 前期准备阶段
- 收集所有相关原型图、交互稿、视觉稿
- 确认产品目标和用户群体
- 整理现有业务流程和痛点
- 确定需求拆解的粒度（功能模块 → 功能点 → 验收标准）

### 2. 原型图分析阶段
#### 2.1 整体结构梳理
- 识别页面布局结构（顶部导航、侧边栏、主要内容区）
- 确定系统的主要功能入口（从侧边菜单或顶部导航）
- 标记当前显示的页面/功能模块

#### 2.2 功能模块识别
- 从侧边菜单提取所有一级功能模块
- 识别当前高亮/选中的功能模块作为重点分析对象
- 为每个功能模块分配唯一编号（如 MOD-001, MOD-002）

#### 2.3 界面元素拆解
对于每个功能模块，按以下维度拆解：
- **导航元素**：面包屑、标签页、步骤条
- **操作区域**：按钮组、搜索框、过滤器、批量操作
- **数据展示**：表格、列表、卡片、图表、时间轴
- **表单元素**：输入框、下拉选择、日期选择器、复选框、单选框
- **反馈元素**：提示信息、加载状态、空状态、错误状态
- **布局特征**：固定头部/底部、滚动区域、分栏布局

#### 2.4 交互行为识别
- 点击行为：按钮点击、链接跳转、展开/收起
- 输入行为：表单填写、搜索过滤、排序切换
- 拖拽行为：排序调整、时间调整、元素移动
- 手势行为（如适用）：滑动切换、 pinch缩放
- 自动行为：页面加载时的默认状态、定时刷新

#### 2.5 数据流向分析
- 标识数据来源：后端API、本地存储、用户输入
- 标识数据去向：页面展示、下一步操作、数据提交
- 识别关键数据实体及其属性
- 标识数据变化触发条件

### 3. 需求文档化阶段
#### 3.1 功能模块定义
使用需求文档模板中的"2.2 详细功能描述"部分，为每个功能模块填写：
- 功能描述：用一句话概括该模块解决什么问题
- 业务流程：描述用户使用该模块的完整路径
- 功能点明细：将UI元素和交互行为转化为可验证的功能点
- 业务规则：从界面限制、验证提示中提取业务约束
- 数据需求：从表单和展示中提取需要的数据字段
- 接口需求：根据数据流向推断需要的API接口

#### 3.2 验收标准制定
将功能点转化为可测试的验收标准：
- 功能验收：每个功能点对应一个或多个验收标准
- 性能验证：针对数据展示量、并发用户数制定性能指标
- 兼容性验证：根据目标用户群体确定测试环境
- 异常情况：考虑网络失败、数据为空、权限不足等场景

### 4. 需求评审与迭代
- 组织开发、测试、UI设计师进行需求评审
- 根据技术可行性调整需求复杂度
- 根据用户反馈补充或修改需求
- 保持需求文档的版本控制和变更追踪

## 二、针对康养钻石湾原型图的具体拆解方法

### 1. 顶级功能模块识别
从侧边菜单可识别以下顶级功能模块：

- MOD-003: 全渠道订单处理

- MOD-005: 冲突预警处理
- MOD-006: 商品与交易中心
- MOD-007: 会员与客户管理

- MOD-009: 系统配置中心

### 2. 以"多维调度日历板"（MOD-002）为例的拆解过程

#### 2.1 功能模块概述
- **功能名称**：多维调度日历板
- **功能描述**：提供多维度（资源类型、所属门店、资源状态、时间范围）的调度日历视图，支持预约的查看、创建、修改和冲突检测
- **主要用户**：调度员、前台接待、资源管理者
- **使用场景**：日常调度工作、资源分配、时间冲突预防

#### 2.2 界面元素与功能点映射

##### 2.2.1 页面导航区
- 面包屑导航：首页 > 统一预约工作台 > 多维调度日历板
  - 功能点：FP-Nav-001 - 显示当前位置的面包屑导航
  - 功能点：FP-Nav-002 - 支持通过面包屑返回上级页面

##### 2.2.2 页面头部操作区
- 页面标题："多维调度日历板"
- 日期导航：上一天/今天/下一天按钮 + 日期显示
  - 功能点: FP-DateNav-001 - 显示当前选中的日期
  - 功能点: FP-DateNav-002 - 上一天按钮：将日期减一天
  - 功能点: FP-DateNav-003 - 下一天按钮：将日期加一天
  - 功能点: FP-DateNav-004 - 今天按钮：将日期重置为当天
- 视图切换：日视图/周视图/月视图按钮
  - 功能点: FP-ViewSwitch-001 - 日视图按钮：切换到日视图模式
  - 功能点: FP-ViewSwitch-002 - 周视图按钮：切换到周视图模式
  - 功能点: FP-ViewSwitch-003 - 月视图按钮：切换到月视图模式
  - 功能点: FP-ViewSwitch-004 - 根据选中视图高亮对应按钮
- 功能按钮：刷新、全屏
  - 功能点: FP-ActionBtn-001 - 刷新按钮：重新加载日历数据
  - 功能点: FP-ActionBtn-002 - 全屏按钮：切换全屏/普通模式显示

##### 2.2.3 过滤面板
- 资源类型下拉选择
  - 功能点: FP-Filter-001 - 显示资源类型下拉选择框
  - 功能点: FP-Filter-002 - 支持选择单个资源类型进行过滤
  - 功能点: FP-Filter-003 - 默认显示"全部资源类型"选项
- 所属门店下拉选择
  - 功能点: FP-Filter-004 - 显示所属门店下拉选择框
  - 功能点: FP-Filter-005 - 支持选择单个门店进行过滤
  - 功能点: FP-Filter-006 - 默认显示"全部门店"选项
- 资源状态下拉选择
  - 功能点: FP-Filter-007 - 显示资源状态下拉选择框
  - 功能点: FP-Filter-008 - 支持选择资源状态（如：全部、可用、维修中、已预订）
  - 功能点: FP-Filter-009 - 默认显示"全部状态"选项
- 时间范围选择器
  - 功能点: FP-Filter-010 - 显示时间范围选择器（开始日期-结束日期）
  - 功能点: FP-Filter-011 - 支持通过日期选择器选择开始和结束日期
  - 功能点: FP-Filter-012 - 默认显示最近一周的时间范围
- 操作按钮组
  - 功能点: FP-Filter-013 - 查询按钮：根据过滤条件查询并显示日历数据
  - 功能点: FP-Filter-014 - 重置按钮：将所有过滤条件重置为默认值

##### 2.2.4 主要内容区 - 时间轴网格
- 资源标签列（左侧固定列）
  - 功能点: FP-Grid-001 - 显示资源名称列表（垂直方向）
  - 功能点: FP-Grid-002 - 资源列支持垂直滚动，时间轴区域水平滚动
  - 功能点: FP-Grid-003 - 资源行高度固定，支持显示完整资源名称
- 时间槽列表（横向时间轴）
  - 功能点: FP-Grid-004 - 显示时间槽标题（如：08:00, 09:00, 10:00...）
  - 功能点: FP-Grid-005 - 时间槽间隔可配置（默认30分钟或1小时）
  - 功能点: FP-Grid-006 - 时间槽支持水平滚动查看更多时间段
- 网格单元格
  - 功能点: FP-Grid-007 - 网格单元格显示对应时间段的预约信息
  - 功能点: FP-Grid-008 - 无预约时显示为可点击的添加预约区域
  - 功能点: FP-Grid-009 - 有预约时显示预约详情（时间、客户、服务类型等）
  - 功能点: FP-Grid-010 - 支持点击网格单元格创建新预约
  - 功能点: FP-Grid-011 - 支持点击已有预约查看/编辑预约详情
  - 功能点: FP-Grid-012 - 预约块根据状态使用不同颜色标识（如：待确认、已确认、已完成）
  - 功能点: FP-Grid-013 - 支持拖拽预约块调整时间（如适用）
  - 功能点: FP-Grid-014 - 网格支持按小时/半小时/15分钟等不同精度显示

#### 2.3 业务规则提取
从界面和交互中可推断以下业务规则：
- BR-001: 同一资源在同一时间段不能有重复预约（防止双订）
- BR-002: 预约开始时间必须早于结束时间
- BR-003: 只能预约未来时间段的资源（不能预约过去时间）
- BR-004: 某些资源类型可能有最小预约时长限制（如：会议室最少1小时）
- BR-005: 超出工作时间的时段应该视觉区分或不可选
- BR-006: 预约创建时需要选择客户、服务类型、联系人等必要信息
- BR-007: 预约修改不应影响已进行中的服务

#### 2.4 数据需求提取
从界面元素可识别以下数据需求：
- DT-001: 资源ID (String) - 唯一标识资源
- DT-002: 资源名称 (String) - 资源显示名称
- DT-003: 资源类型 (String) - 如：会议室、理疗室、健身器械
- DT-004: 所属门店ID (String) - 资源所属门店
- DT-005: 资源状态 (Enum) - 可用/维修中/已预订/停用
- DT-006: 预约ID (String) - 唯一标识预约
- DT-007: 预约标题 (String) - 预约主题
- DT-008: 客户ID (String) - 预约对应的客户
- DT-009: 客户名称 (String) - 客户显示名称
- DT-010: 预约开始时间 (DateTime) - 预约起始时间
- DT-011: 预约结束时间 (DateTime) - 预约结束时间
- DT-012: 服务类型 (String) - 预约的服务项目
- DT-013: 联系人 (String) - 预约联系人姓名
- DT-014: 联系电话 (String) - 联系人电话
- DT-015: 备注 (Text) - 预约备注信息
- DT-016: 预约状态 (Enum) - 待确认/已确认/已完成/已取消/已过期

#### 2.5 接口需求推断
根据数据流向可推断需要的API接口：
- API-001: GET /api/resources - 获取资源列表（支持按类型、门店、状态过滤）
- API-002: GET /api/appointments - 获取预约列表（支持按资源、时间范围、状态过滤）
- API-003: POST /api/appointments - 创建新预约
- API-004: GET /api/appointments/{id} - 获取单个预约详情
- API-005: PUT /api/appointments/{id} - 更新预约信息
- API-006: DELETE /api/appointments/{id} - 删除预约
- API-007: GET /api/resource-types - 获取资源类型字典（用于过滤下拉）
- API-008: GET /api/stores - 获取门店列表（用于过滤下拉）
- API-009: GET /api/appointment-statuses - 获取预约状态字典
- API-010: GET /api/resource-statuses - 获取资源状态字典

## 三、需求拆解的最佳实践建议

### 1. 原型图分析技巧
- **分层观察**：先看整体布局，再看功能区，最后看具体控件
- **对比参照**：将不同样-state的原型图进行对比（如：空状态vs数据状态）
- **交互痕迹**：注意原型图中的hover状态、选中状态、禁用状态提示
- **注释利用**：善用原型图中的设计说明和交互备注

### 2. 需求表达原则
- **可验证性**：每个需求点都应该能够设计测试用例来验证
- **无歧义性**：避免使用"可能"、"应该"等模糊表述
- **完整性**：覆盖正常流程、异常流程、边界条件
- **可追溯性**：需求编号要清晰，便于变更追踪和测试关联

### 3. 常见陷阱避免
- **过度设计**：不要在需求中体现具体的实现细节（如具体使用什么技术）
- **遗漏边界**：注意空数据状态、大数据量状态、权限不足状态
- **假设用户行为**：所有功能点都应该能从原型图中直接或间接推导得出
- **忽略非功能**：性能、安全、兼容性等非功能需求也需考虑

### 4. 工具与协作建议
- **标注工具**：使用截图标注工具在原型图上直接标注功能点
- **需求追踪**：建立需求原型图 ↔ 功能点 ↔ 测试用例的追踪矩阵
- **版本管理**：原型图和需求文档要保持版本同步
- **评审机制**：定期进行需求评审会，收集不同角色的反馈

## 四、附件：康养钻石湾需求拆解示例

### 示例：多维调度日历板 - 时间轴网格功能点

**功能点编号**：FP-Grid-007  
**功能点名称**：网格单元格预约信息展示  
**功能点描述**：在时间轴网格的交点位置，显示对应时间段已有预约的详细信息  
**输入数据**：预约列表（包含预约ID、标题、开始时间、结束时间、客户名称、服务类型、状态）  
**展示规则**：
- 预约块水平跨度对应预约的时间跨度（开始时间到结束时间）
- 预约块垂直位置对应资源所在行
- 预约块颜色根据状态不同：待确认（#FFC107）、已确认（#4CAF50）、已完成（#2196F3）、已取成（#F44336）
- 预约块内部显示：[预约标题] - [客户名称] （如：理疗预约 - 张三）
- 当预约时间跨度少于最小显示宽度时，显示折叠状态并提供tooltip查看完整信息
**验收标准**：
- AC-Grid-007-01: 给定预约数据，系统能正确计算并显示预约块的位置和尺寸
- AC-Grid-007-02: 不同状态的预约使用对应的颜色标识
- AC-Grid-007-03: 预约块内部信息格式正确：[标题] - [客户名称]
- AC-Grid-007-04: 当预约时间跨度小于30分钟时，显示折叠状态并hover时显示完整tooltip
- AC-Grid-007-05: 点击预约块能正确触发查看预约详情的操作

通过以上方法，产品经理可以系统地从原型图中提取出完整、清晰、可验证的需求，为后续的开发任务拆解提供坚实的基础。
