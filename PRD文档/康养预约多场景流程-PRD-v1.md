## 1. 产品介绍

### 1.1. 背景与目标

当前康养业务存在多来源获客、多路径交付、多分支退改，跨系统协同复杂（CRM/HIS/PMS/权益/支付/调度/企业微信），导致预约链路标准不统一、异常处理口径不一致、跨部门协作追踪困难。

本产品目标是构建“统一预约调度中心”，对康养全渠道业务进行标准化编排，实现“购买-交付-退改”全链路闭环：

1. 统一入口：覆盖医转店/店转医/线上直营/自然到店/权益客户/团体客户等场景。
2. 统一状态：订单、预约、履约、退改状态统一可追踪。
3. 统一规则：资源校验、支付核销、退改分级、SLA规则统一执行。
4. 统一协同：管家、前台、礼宾、客房、健康评估师、医生、财务多角色协同。
5. 统一运营：形成可量化的转化、履约、满意度、复购、成本与ROI指标体系。

### 1.2. 核心功能

- **多场景预约受理**：支持客户自助、销售代客、管家代客、医生/评估师代客、第三方渠道接入。
- **资源实时校验与锁定**：房型/医疗号源/项目容量/接驳车辆联合校验，失败即时分流。
- **支付与权益一体化**：定金/尾款/全额/权益核销统一能力，自动生成订单与预约单。
- **预抵与交付编排**：自动推送问卷、生成行程单、派单至多部门并追踪完成状态。
- **交付确认与结算**：核销、分账、对账、结算全流程闭环。
- **退改分级决策**：按核销状态、交付状态、时间窗口、规则命中自动分流至自动处理或人工审批。
- **运营中台能力**：客户全景、预警、工单、生命周期运营、离店分析、智能排班与BI分析。

### 1.3. 目标用户与应用场景

| 目标用户 | 应用场景 |
| -------- | -------- |
| 市场销售/康养销售 | 老客户再转化、康养老客户二次转化、店转医推荐成交 |
| 商务经理/团体领队 | 企业合作/第三方合作团体预约与批量履约 |
| 客户/客户家属 | 自主预约、在线支付、行程查看、接驳可视、退改申请 |
| 康养管家/前台/礼宾/客房/餐饮 | 预抵沟通、接驳到店、入住交付、任务反馈 |
| 医生/健康评估师 | 医康方案确认、医疗加项识别、代客预约 |
| 运营/财务/管理层 | 订单履约、异常治理、分账对账、经营分析与预警 |

覆盖场景（与流程图及Sheet保持一致）：

1. 医转店-老客户再转化
2. 医转店-商务拓展客户（企业/第三方）
3. 医转店-线上直营获客（APP/小程序）
4. 店转医-老市场带新客户
5. 店转医-自然到店
6. 店转医-多渠道获客（OTA/旅行社）
7. 店转医-权益性客户
8. 店转医-康养老客户转化
9. 后台运营管理（全渠道通用）

### 1.4. 名词解释

- **预约单**：客户完成预约动作后生成的履约凭证。
- **订单**：支付或权益核销完成后形成的结算主单。
- **权益核销**：会员权益额度的冻结、扣减与回退动作。
- **医康融合预约**：同一次业务中同时预约康养与医疗资源。
- **预抵行程单**：入住前生成的包含接驳、入住、评估、项目安排的执行计划。
- **部分交付**：订单中部分项目已履约、部分未履约。

## 2. 系统架构与流程

### 2.1. 系统架构流程图（文字版）

1. 渠道接入层：APP/小程序、企微、电话、销售工作台、第三方渠道（OTA/旅行社/企业）。
2. 预约编排层：资源校验、规则引擎、支付核销、预约生成、行程编排。
3. 交付协同层：管家工作台、接驳调度、前台入住、客房餐饮任务、健康评估与医生协作。
4. 结算与风控层：核销、分账、对账、结算、退改策略引擎、审批流。
5. 数据中台层：客户360、标签画像、生命周期运营、BI分析、审计日志。

### 2.2. 数据流说明

第一部分：预约前置与订单生成

1. 多渠道请求进入预约中心，统一做客户识别与建档。
2. 完成权益校验/资源校验（房态、号源、项目容量、车辆）。
3. 发起支付或权益核销，生成订单与预约单。
4. 状态进入“待履约”，同步至PMS/HIS/CRM/调度系统。

第二部分：履约交付

1. 自动指派管家，建立客户沟通触点，发送问卷和预抵行程单。
2. 系统自动派发部门任务，接驳、入住、评估、项目执行按计划推进。
3. 交付过程实时上报，异常触发预警与应急流程。
4. 交付完成后进行核销确认，状态进入“已交付/部分交付”。

第三部分：退改与结算

1. 客户发起退改申请，策略引擎基于核销状态、时间窗口、交付完成度分流。
2. 命中自动规则则系统自动退款/回退权益；否则进入人工审批。
3. 完成分账、对账、财务结算，订单闭环归档。

### 2.3. 多场景流程（对应预约多场景流程图）

1. 图1（总览）：产品资源配置完成后进入“资源可约判断→购买→交付→退改”。
2. 图2（线上第三方平台）：平台展示可约资源，客户下单支付，交付后按规则退改。
3. 图3（旅行社渠道）：团体合同与名册前置，预约同步到调度中心后交付与退改分流。
4. 图4（商务团）：签约与批量预约后，重点在团体接待与集中退改处理。
5. 图5（直营/门店）：导购触发预约，系统自动校验并分流到交付，再进入退改判定。
6. 图6（退改分级）：按“是否核销/是否交付/是否部分交付/是否超时”执行自动或人工策略。

### 2.4. 全链路统一状态定义

| 状态码 | 状态名称 | 说明 |
| ------ | -------- | ---- |
| 10 | 待校验 | 请求已创建，等待资源/权益校验 |
| 20 | 待支付/待核销 | 可约资源已确认，等待支付或权益冻结 |
| 30 | 已预约待交付 | 订单与预约单已生成，等待履约 |
| 40 | 交付中 | 已到店或服务进行中 |
| 50 | 已完成 | 全部交付完成且核销确认 |
| 55 | 部分交付 | 部分项目完成，允许分项退改 |
| 60 | 退改处理中 | 已发起退改，规则或审批处理中 |
| 70 | 已退改 | 退款/权益回退完成 |
| 80 | 已关闭 | 结束态（完成、拒绝、超时关闭） |
| 90 | 异常中断 | 系统异常或人工中断，待补偿 |

## 3. 功能详述

### 3.1. 多场景预约受理与入口治理

- **功能描述**：统一接收8类业务场景与第三方渠道预约请求，做渠道标准化映射。
- **输入内容**：客户身份、渠道来源、权益信息、预约诉求、时间与地点。
- **核心流程**：
  1. 渠道识别与路由（来源1~8 + 运营后台）。
  2. 去重防重（同客户同时间窗重复请求拦截）。
  3. 预约意图拆解（住宿/康养项目/医疗项目/接驳）。
- **输出内容**：标准化预约请求单。
- **异常处理**：渠道数据缺失时进入补录队列并标记风险等级。

### 3.2. 客户识别、建档、授权与数据同步

- **功能描述**：完成客户主档建立、授权签署、医康数据双向脱敏同步。
- **核心流程**：
  1. 新老客识别与客户360拉取。
  2. 电子授权签署与状态记录。
  3. 医疗/非医疗数据按字段级脱敏同步。
- **系统要求**：全量审计日志、字段脱敏策略、失败重试与告警。
- **输出内容**：完整客户档案、授权状态、同步回执。

### 3.3. 资源校验、下单、支付与权益核销

- **功能描述**：联合校验房型/号源/项目/车辆资源，完成支付或权益冻结。
- **核心流程**：
  1. 资源可约性判断；不可约则给出替代建议。
  2. 支持定金、全额、混合支付、权益抵扣。
  3. 生成订单、预约单、任务码并同步调度中心/PMS/HIS。
- **异常处理**：支付失败重试、权益不足补差、锁资源失败自动释放。

  1. 自动/人工管家指派（优先原管家）。
  2. 预抵沟通与健康问卷推送（可按90天有效期判定是否重发）。

- **SLA**：管家首次联系 <= 24小时，紧急群消息响应 <= 15分钟。

### 3.6. 订单核销、分账、对账与结算

### 3.7. 退改受理与分级决策

- **功能描述**：依据图6规则执行“自动处理优先、人工审批兜底”。
- **关键判断条件**：
  1. 是否已核销/已交付。
  2. 是否部分交付。
  3. 是否超出可退改时间窗。
  4. 是否触发人工介入条件（高金额、跨系统冲突、异常订单）。
- **自动处理动作**：退权益、退款、释放资源、回写状态、通知客户。
- **人工处理动作**：审批、补录、补扣、财务记账、客诉说明。

## 4. 数据库设计

### 4.1. 核心表清单

1. `customer_profile`：客户主档与标签。
2. `customer_authorization`：授权协议与有效期。
3. `channel_booking_request`：多渠道预约请求原始单。
4. `booking_order`：订单主表。
5. `booking_item`：订单子项（住宿/项目/医疗/接驳）。
6. `resource_lock_record`：资源锁定与释放记录。
7. `service_delivery_record`：交付执行记录。
8. `verification_record`：核销记录（全额/部分）。
9. `refund_change_request`：退改申请单。
10. `refund_change_decision`：退改策略命中与审批记录。
11. `payment_transaction`：支付与退款流水。
12. `settlement_statement`：分账与对账单。
13. `dispatch_task`：接驳与部门任务工单。
14. `system_event_log`：关键系统事件与审计日志。

### 4.2. 关键字段建议（示例）

`booking_order`

- `order_id`：订单号（唯一）
- `customer_id`：客户ID
- `channel_code`：渠道编码（S1~S8/OPS）
- `order_status`：统一状态码（10~90）
- `delivery_status`：未交付/交付中/已交付/部分交付
- `payment_status`：未支付/已支付/部分退款/已退款
- `verification_status`：未核销/已核销/部分核销
- `total_amount` / `paid_amount` / `refunded_amount`
- `sla_deadline`：关键时限
- `risk_level`：风险等级
- `created_time` / `updated_time`

`refund_change_request`

- `rc_id`：退改单号
- `order_id`：关联订单号
- `rc_type`：退款/改期/改项目
- `trigger_source`：客户发起/系统触发/人工补录
- `rule_hit_code`：策略命中编码
- `auto_process_flag`：是否自动处理
- `manual_approver`：人工审批人
- `rc_status`：待处理/处理中/已完成/已拒绝
- `reason_code`：原因编码

### 4.3. 状态流转约束

1. 订单必须先完成资源校验后才可进入支付/核销状态。
2. 已全量核销且已全量交付订单不可整单退款，仅允许规则内补偿。
3. 部分交付订单仅允许对子项退改，需保留履约子项结算结果。
4. 任一跨系统同步失败需记录补偿任务，不得直接跳过。

### 4.4. 报表指标模型

- 转化漏斗：线索 -> 预约 -> 到店 -> 交付 -> 复购。
- 履约质量：交付准时率、异常率、客诉率、补偿率。
- 经营效能：客单价、复购率、渠道ROI、物料成本效率。
- 财务健康：退款率、对账差异率、分账及时率。
